# 项目技术文档

## 1. 项目概述

本项目是一个前后端分离的在线学习管理平台，包含以下核心业务模块：

- 登录认证
- 工作台统计与图表展示
- 课程管理
- 学生管理
- 学习总结 Markdown 展示
- 前端路由守卫与登录态持久化
- 后端静态托管前端打包产物

从项目体量上看，它不是一个超大型系统，但它很好地覆盖了中后台项目中最常见、也最适合面试展开讨论的能力点：

- 前后端分离架构
- 统一接口封装
- 登录鉴权与权限边界
- 状态管理
- 列表页增删改查
- 表单校验
- 图表可视化
- SQLite 数据建模
- 服务端数据聚合
- Markdown 渲染与静态资源访问

如果把它当成一个学习项目，它的价值不只是“能跑”，更在于它把一个完整管理后台从页面、状态、接口、数据库到部署串起来了。

## 4. 系统整体架构

## 4.1 架构模式

本项目采用典型的前后端分离架构：

1. 前端负责页面渲染、交互、状态管理、表单校验。
2. 后端负责认证、数据查询、数据写入、统计聚合、静态资源读取。
3. 数据库存储业务实体与统计基础数据。
4. 前端通过 HTTP 调用后端统一接口。

可以把整个请求链路理解成：

```text
用户操作
  -> React 页面组件
  -> Zustand store
  -> api.ts
  -> request.ts(Axios 拦截器)
  -> Koa 路由
  -> SQLite
  -> 返回统一 JSON
  -> store 更新状态
  -> 页面重新渲染
```

## 5. 前端架构设计

## 5.1 前端启动流程

入口文件是 `client/src/main.tsx`，主要做两件事：

1. 获取根节点 `#root`
2. 以 `StrictMode` 挂载 `App`

`App.tsx` 负责更高一层的应用装配：

1. 使用 `ConfigProvider` 配置 Ant Design 主题
2. 引入全局样式
3. 挂载 `AppRouterProvider`

这说明作者把“应用主题”和“应用路由”作为最顶层能力集中处理，而不是散落在页面中。

---

## 5.2 路由设计

前端路由定义在 `client/src/router/routes.tsx`。

路由结构如下：

- `/login`：登录页
- `/dashboard`：工作台
- `/courses`：课程管理
- `/students`：学生管理
- `/summary`：学习总结

其中 `/` 不是直接渲染页面，而是使用 `ProtectedLayoutElement` 作为受保护布局，再由 `Outlet` 承载子页面。

这是一种很常见也很合理的后台系统设计：

- 登录页与业务页分离
- 业务页共用一个统一布局
- 路由守卫放在布局层，而不是每个页面各写一遍

---

## 5.3 路由守卫实现

路由守卫逻辑集中在 `client/src/router/page-elements.tsx`。

### 登录页路由判断

`LoginRouteElement` 的核心逻辑：

- 如果 `authLoading` 为真，说明应用正在校验登录态，此时显示全屏 loading
- 如果已经有 `token` 且 `user` 存在，说明当前用户已登录，直接跳转到 `/dashboard`
- 否则展示登录页

### 受保护布局判断

`ProtectedLayoutElement` 的核心逻辑：

- 初始化阶段先显示 loading
- 如果没有合法登录态，则跳回 `/login`
- 如果已登录，则渲染 `AppShell`

### 为什么这么设计

这是一个面试中经常会被问到的点：

“为什么不用每个页面单独判断是否登录？”

答案是：

- 逻辑集中，避免重复
- 所有业务页面共享同一套判断逻辑
- 更容易维护导航、布局、错误展示
- 更符合 React Router 的嵌套路由设计方式

---

## 5.4 登录态持久化

登录态持久化由 `client/src/auth.ts` 与 `client/src/stores/auth-store.ts` 共同完成。

### 本地持久化内容

- token 存在 `course_admin_token`
- user 信息存在 `course_admin_user`

### 冷启动恢复流程

应用启动时：

1. `AppRouterProvider` 调用 `initializeAuth()`
2. 从 localStorage 读取 token
3. 如果没有 token，直接标记为“初始化完成但未登录”
4. 如果有 token，调用 `/api/auth/me`
5. 如果接口成功，恢复用户信息
6. 如果失败，清理登录态和所有 store

### 为什么要请求 `/auth/me`

因为本地只有 token，并不能证明它一定合法：

- token 可能过期
- token 可能被篡改
- 用户可能已被删除

所以前端不能只看 localStorage，就必须请求服务端确认。

### initializePromise 的意义

`auth-store.ts` 中使用了一个 `initializePromise`：

- 目的是防止多个组件在冷启动阶段重复触发 `/auth/me`
- 这是一个小但很实用的并发控制技巧

这类实现很适合面试时展开：

- 如何避免重复请求
- 如何处理初始化竞争条件
- 如何让多个组件共享一次异步结果

## 5.5 状态管理设计

前端状态管理使用 Zustand，而不是 Redux。

### 为什么 Zustand 适合这个项目

这个项目的状态规模属于中小型管理后台：

- 登录状态
- 列表页状态
- 表单弹窗状态
- 加载态
- 错误态

它不需要 Redux 那样严格的 action/reducer 模式，因此 Zustand 更轻量、学习成本更低、样板代码更少。

### 各个 store 的职责

#### 1. auth-store

负责：

- 登录
- 登出
- 冷启动登录态校验
- 全局错误信息

#### 2. dashboard-store

负责：

- 工作台数据加载
- 图表与统计数据状态

#### 3. course-store

负责：

- 课程列表数据
- 分类辅助数据
- 分页、搜索、筛选、排序
- 弹窗开关
- 新增/编辑/删除/发布

#### 4. student-store

负责：

- 学生列表数据
- 班级和课程支持数据
- 分页、搜索、筛选
- 新增/编辑/删除

#### 5. summary-store

负责：

- Markdown 内容加载

## 5.6 Store 设计细节

### 查询状态统一收敛

课程和学生列表都采用了统一的 `query` 对象，包含：

- keyword
- page
- pageSize
- 以及各自模块的筛选条件

好处：

- 列表查询参数集中管理
- 刷新时直接使用当前 query
- 分页、搜索、筛选逻辑统一
- 容易序列化到 URL，方便未来扩展

### draftKeyword 的意义

课程页和学生页都没有直接把输入框值绑定到 query.keyword，而是单独维护 `draftKeyword`。

原因：

- 用户输入过程中不应该每敲一个字就请求接口
- 只有在按回车或点击搜索时，才把草稿值提交到真正查询参数

这是一个很常见的“输入态”和“已提交态”分离设计。

### reset-registry 的意义

`client/src/stores/reset-registry.ts` 用来收集所有 store 的重置函数。

作用：

- 退出登录时清空所有业务状态
- token 失效时同步清理页面状态
- 避免新用户看到旧用户的数据残留

这是一种很实用的全局状态清理方案。

## 5.7 页面层设计

页面组件的职责很清晰：

- 读取 store 状态

- 调用 store 方法
- 组织 UI 结构

页面并不直接操作 axios，也不直接处理数据库语义，这说明分层是比较健康的。

### 课程页设计

课程页包含：

- 查询区
- 表格区
- 分页区
- 表单弹窗

这是典型后台 CRUD 页面的标准形态。

这让项目不仅有传统管理后台能力，还有一定的内容渲染能力。

## 5.9 图表系统设计

图表相关代码位于 `client/src/components/echarts/`。

### ChartContainer 的作用

`chart-core.tsx` 封装了 ECharts 实例生命周期：

- 初始化图表实例
- 绑定 resize 事件
- 组件卸载时销毁实例
- option 更新时调用 `setOption`

### 为什么要单独封装 ChartContainer

如果每个图表组件都自己写 `echarts.init()`：

- 重复代码会很多
- 生命周期容易写错
- 容易出现内存泄漏

所以封装容器是合理的抽象。

### 为什么使用 notMerge: true

在切换不同数据或配置时，ECharts 默认可能会合并旧配置，导致：

- 旧 series 残留
- legend 错误
- 坐标轴配置冲突

因此这里使用 `notMerge: true` 每次完整覆盖，保证渲染稳定。

### 各图表职责

- `CourseEnrollmentChart`：柱状图，展示课程选课人数排名
- `LearningActivityChart`：折线图，展示最近 7 天学习人数与学习时长
- `StudentStatusChart`：环形图，展示学生状态分布
- `CourseCategoryChart`：环形图，展示课程分类分布

### 图表数据为什么由后端聚合

如果前端自己聚合统计：

- 需要拿更多原始数据
- 网络开销更大
- 前端逻辑更复杂

所以把统计放在后端更符合职责边界。

## 5.10 Markdown 渲染器设计

`client/src/markdown.tsx` 是这个项目一个很有意思的亮点。

它没有引入 `marked`、`markdown-it`、`react-markdown` 等库，而是自己实现了一个轻量级 Markdown 解析器。

### 支持的语法

- 标题
- 段落
- 有序列表
- 无序列表
- 代码块
- 图片
- 链接
- 行内代码
- 加粗

### 实现思路

#### 1. 先解析成 Block

代码先把文本转成中间结构：

- heading
- paragraph
- list
- code
- image

再根据 Block 类型渲染为 React 元素。

这是一个非常好的分层：

- 解析逻辑和渲染逻辑分离
- 后续扩展更容易

#### 2. 行内语法用正则扫描

`parseInline()` 使用一个大正则解析：

- `**加粗**`
- `` `代码` ``
- `![图片](url)`
- `[链接](url)`

#### 3. 代码高亮自己实现

通过 `tokenizeLine()` 和 `getHighlightPatterns()` 实现简单高亮：

- 关键字
- 字符串
- 注释
- 数字
- JSX 标签

这不是一个完整的语法高亮器，但对当前项目的展示需求已经足够。

### 这部分可能被面试官追问什么

常见追问包括：

- 为什么不用现成 Markdown 库？
- 自己实现 Markdown 的边界是什么？
- 这种实现有什么风险？

可以这样回答：

- 该项目规模较小，只需要支持有限语法
- 自写实现便于学习原理，也减少依赖
- 缺点是功能有限，不支持复杂嵌套语法，也需要注意 XSS 风险

## 6. 后端架构设计

## 6.1 Koa 服务入口

后端入口在 `server/src/index.js`。

启动过程：

1. 初始化数据库
2. 注册 CORS
3. 注册 bodyParser
4. 注册全局错误处理中间件
5. 挂载各业务路由
6. 最后处理前端静态资源和 SPA 回退

### 为什么静态托管放最后

因为：

- `/api/*` 请求必须优先交给接口路由
- 只有非 API 的 GET 请求才应该进入前端静态资源逻辑

这也是一个典型 Koa 中间件顺序问题，面试很喜欢问。

---

## 6.2 统一响应结构

后端所有 JSON 响应都使用统一格式：

```json
{
  "code": 0,
  "msg": "success",
  "data": {}
}
```

失败时：

```json
{
  "code": 400,
  "msg": "错误描述",
  "data": null
}
```

### 这样设计的好处

- 前端拦截器可以统一处理
- 页面层不用每次判断不同接口格式
- 成功与失败结构一致
- 更容易扩展日志和监控

---

## 6.3 鉴权设计

鉴权中间件在 `server/src/middleware/auth.js`。

### 鉴权流程

1. 从请求头中读取 `Authorization`
2. 解析 `Bearer token`
3. 使用 `jsonwebtoken.verify()` 校验
4. 成功后把用户信息挂到 `ctx.state.user`
5. 失败则直接返回 401

### 为什么用户信息放到 ctx.state.user

因为这是 Koa 常见约定：

- 中间件处理后的共享上下文
- 后续路由可以直接读取
- 不污染 request body 或 query

---

## 6.4 认证模块设计

认证路由在 `server/src/routes/auth.js`。

### 登录

流程：

1. 校验用户名密码是否存在
2. 从数据库查询用户
3. 使用 `bcrypt.compareSync` 校验密码
4. 使用 `jwt.sign` 签发 token
5. 返回 token 和去掉 password 的 user 信息

### 为什么登录失败统一提示

项目对“用户名不存在”和“密码错误”都返回：

`用户名或密码错误`

原因：

- 避免暴露用户名是否存在
- 降低被枚举账号的风险

这是一个典型安全设计点。

### /me 接口的作用

它不是给页面显示头像这么简单，而是前端恢复登录态的关键接口。

如果面试官问：

“为什么有 token 还要请求 `/me`？”

可以回答：

- token 只证明“客户端保存了一个字符串”
- `/me` 才能证明这个 token 当前是否合法，并且能拿到最新用户信息

## 6.5 课程模块设计

课程路由在 `server/src/routes/courses.js`。

### 支持的能力

- 列表查询
- 分类查询
- 单项详情
- 新增
- 编辑
- 删除
- 状态切换

### 列表查询实现

列表接口支持：

- keyword 搜索课程名或讲师
- status 筛选
- category 筛选
- sortField + sortOrder 排序
- page + pageSize 分页

### SQL 设计亮点

#### 1. 参数化查询

使用 `where + params` 的方式拼接：

- 灵活
- 安全
- 避免 SQL 注入

#### 2. 排序白名单

排序字段只能从这些值里选：

- student_count
- lesson_count
- created_at
- name

这是为了防止用户把任意字符串拼进 `ORDER BY`。

### 新增与编辑设计

编辑接口使用“未传值则回退 existing”的方式：

- 支持部分更新
- 保证未修改字段不会丢失

### 状态切换设计

课程状态只有两种：

- published
- draft

因此后端直接做二元切换，逻辑简单清晰。

这是整个后端里最值得仔细学习的模块之一，因为它包含：

- 查询

### 查询能力

支持：

- keyword 搜姓名或学号
- className 筛选
- status 筛选
- courseId 按课程过滤
- 分页

### course_ids 的存储方式

学生和课程是多对多关系，但项目没有单独建中间表，而是直接在 `students.course_ids` 中存 JSON 数组。

例如：

```json
[1, 3, 5]
```

### 这种设计的优点

- 数据结构简单
- 开发速度快
- 对作业项目足够

### 这种设计的缺点

- SQL 查询关系不方便
- 无法高效 join
- 按课程过滤时需要先查再内存过滤
- 扩展性不如中间表

### 为什么仍然可以接受

因为这个项目规模较小，主要目的是学习和演示，权衡开发成本后这是合理选择。

这类“工程取舍”很适合面试聊。

### parseStudentPayload 的作用

这是学生路由的核心函数之一，它把新增和编辑的字段校验统一到了一个地方。

它负责：

- 文本清洗
- 回退旧值
- course_ids 归一化
- 学号格式校验
- 手机号格式校验
- 邮箱格式校验
- 状态合法性校验
- 至少选择一门课程校验

### 为什么把校验集中在这里

因为这样可以让：

- POST 和 PUT 共用同一套规则
- 逻辑不重复
- 更容易维护

### updateCourseCounts 的意义

学生的增删改会影响课程的选课人数。

项目没有每次实时 join 统计，而是把 `courses.student_count` 作为冗余字段维护。

好处：

- 课程列表查询快
- 工作台图表查询快

缺点：

- 写操作后需要同步更新冗余数据

这是典型的“用写复杂度换读性能”的设计思路。

## 9. 请求封装与错误处理

## 9.1 request.ts 的职责

前端所有请求都经过 `client/src/utils/request.ts`。

它做了三件关键事情：

1. 自动注入 token
2. 统一判断 `ApiEnvelope.code`
3. 统一抛出 `Error`

### 为什么统一抛 Error

这样 store 或页面层可以统一：

- `try/catch`
- 用 `appErrorMessage()` 处理
- 再把错误交给全局错误提示或字段错误提示

## 9.2 全局错误处理策略

大多数业务错误最终进入 `auth-store.globalError`。

由布局层 `AppShell` 统一显示 `Alert`。

好处：

- 页面少写重复错误 UI
- 交互一致
- 用户更容易感知系统状态

当然，字段级错误仍然保留在表单里，比如：

- 学号已存在

这种错误既可以显示在全局，也可以回填到具体字段上。

## 10. 前后端数据流详解

这里以几个典型场景说明整个系统如何工作。

## 10.1 登录流程

```text
用户输入账号密码
  -> LoginPage 提交
  -> auth-store.handleLogin()
  -> api.login()
  -> request.ts 发送 POST /api/auth/login
  -> Koa auth.js 登录校验
  -> 返回 token + user
  -> setAuth 持久化到 localStorage
  -> auth-store 写入 token/user
  -> 路由守卫跳转到 /dashboard
```

---

## 10.2 冷启动恢复登录态

```text
刷新页面
  -> AppRouterProvider 调用 initializeAuth()
  -> 从 localStorage 取 token
  -> 调用 /api/auth/me
  -> 服务端校验 JWT
  -> 返回当前用户信息
  -> 写回 auth-store
  -> 路由恢复到业务页面
```

---

## 10.3 课程列表查询

```text
用户选择筛选条件
  -> 页面调用 updateQuery()
  -> course-store.refreshList()
  -> api.fetchCourses(query)
  -> GET /api/courses
  -> 后端拼 where/orderBy/page
  -> SQLite 返回分页数据
  -> store 更新 data
  -> Table 重新渲染
```

---

## 10.4 学生新增流程

```text
用户填写表单
  -> antd Form 同步校验
  -> 学号失焦异步唯一性校验
  -> 点击保存
  -> student-store.submitForm()
  -> POST /api/students
  -> 后端 parseStudentPayload 校验
  -> 写入 students
  -> updateCourseCounts()
  -> 返回新学生
  -> 刷新列表
```

- 后端校验是为了数据安全和最终可信性
- 前端永远不能替代后端校验

## 11.3 为什么用 JWT

可回答：

## 11.5 为什么要维护 courses.student_count 冗余字段

可回答：

- 因为课程列表和工作台都频繁展示选课人数
- 每次动态聚合会增加查询复杂度
- 这个项目写操作不频繁，适合“写时更新冗余字段”

同时也要说明风险：

- 如果更新漏掉，会产生数据不一致
- 更成熟系统可能用数据库触发器、事件驱动或定时校准

- 减少前端拿原始数据后的二次加工逻辑

## 16. 进阶技术细节拆解

这一章重点不是重复介绍模块，而是把项目中容易被忽略、但在面试里很容易被继续追问的技术细节单独拎出来讲。

## 16.1 为什么前端 API 要分成 `api.ts` 和 `request.ts`

很多初学者会把所有请求都直接写在页面里，例如：

```ts
useEffect(() => {
  axios.get('/api/courses').then(...)
}, [])
```

这种写法在 demo 阶段没问题，但一旦页面增多，问题会很快暴露：

- token 注入逻辑会到处重复
- 错误处理不统一
- 请求类型不清晰
- 页面组件过于臃肿

本项目把请求层拆成两层：

### 第一层：`request.ts`

职责是“基础设施层”：

- 创建 Axios 实例
- 设置 `baseURL`
- 设置超时
- 注入 token
- 统一解析响应格式
- 统一处理 401 和错误消息

### 第二层：`api.ts`

职责是“业务接口层”：

- 声明接口名称
- 管理 URL
- 管理查询参数序列化
- 对返回值做类型约束

这种拆法的好处是：

- 页面只关心“我要什么数据”
- 不关心“这个请求怎么鉴权、怎么解包、怎么处理错误”

从架构角度讲，这相当于把“通信协议细节”从业务层抽离了出去。

---

## 16.2 为什么全局错误放在 `auth-store` 里

从语义上看，把 `globalError` 放在 `auth-store` 里好像有点奇怪，因为很多错误并不属于认证模块。

但从项目演进角度看，这是一种“工程上够用且集中”的方案。

原因如下：

### 1. 登录态是整个应用的最顶层状态

布局、导航、路由守卫都围绕登录态组织，因此把全局错误也放在一个顶层 store 里，可以减少跨 store 通信复杂度。

### 2. 布局层天然依赖 auth-store

`AppShell` 已经依赖用户信息和登出行为，所以直接读取 `globalError` 最方便。

### 3. 错误展示位置本身也是全局性的

既然错误被展示在布局顶部，而不是某个独立业务页局部区域，那么它从某种意义上就是一个“全局 UI 状态”。

### 这种做法的边界

如果项目继续变大，更推荐把它抽成：

- `app-store`
- `ui-store`
- `notification-store`

这样语义更清晰。

所以面试时可以回答：

- 当前项目规模下放在 `auth-store` 是一种轻量实现
- 更大规模系统会考虑单独的全局 UI 状态仓库

---

## 16.3 为什么使用 `useShallow`

课程页和学生页在读取 Zustand store 时使用了：

```ts
useShallow(...)
```

它的主要作用是减少不必要的重新渲染。

### 背景

如果组件直接订阅一个对象：

```ts
const state = useStore((s) => ({
  a: s.a,
  b: s.b,
}));
```

每次 store 更新都会返回一个新对象引用，即使 `a` 和 `b` 没变，也可能触发组件重新渲染。

### `useShallow` 的作用

它会做浅比较：

- 如果字段值没变，就不触发重渲染
- 如果字段确实变了，才更新页面

### 这在本项目中的价值

列表页状态很多：

- loading
- data
- formOpen
- editingId
- draftKeyword
- categories/classes/courses

如果不做选择性订阅，性能虽然不至于崩，但会产生很多无意义重渲染。

---

## 16.4 为什么表单弹窗使用 `destroyOnHidden`

课程页和学生页的弹窗都配置了：

- `destroyOnHidden`

它的价值在于：关闭弹窗后销毁内部内容。

### 好处

- 下次打开重新走表单初始逻辑
- 避免隐藏状态下保留旧输入
- 避免部分受控/非受控组件状态错乱

### 风险

如果弹窗里有特别复杂的内容，每次重新挂载可能有性能成本。

但在本项目里，表单规模适中，这个取舍是合理的。

## 16.5 为什么课程和学生列表不使用 Table 自带分页

项目选择关闭 Table 内建分页，而使用自己的 `PaginationBar`。

### 原因

#### 1. 统一所有页面的分页交互

课程页、学生页都使用相同的分页行为：

- 页码切换
- pageSize 切换
- 总条数展示

统一成一个组件更利于维护。

#### 2. 逻辑收敛在 store 的 `query`

分页本质上是查询参数的一部分，不应该由表格组件自己“悄悄管理”。

#### 3. UI 样式统一

通过自定义分页栏，可以让视觉和交互跟整个后台一致。

### 面试答法

如果被问“为什么不用现成分页”，可以说：

- 不是不能用，而是为了更统一地控制查询参数和 UI 行为
- 本项目希望把分页作为查询状态的一部分显式管理

## 16.6 表单校验的层次设计

学生表单是这个项目中校验最完整的一个地方。

它有三层校验：

### 第一层：基础同步校验

例如：

- 必填
- 最大长度
- 正则格式

这类校验特点：

- 快
- 无需请求后端
- 能第一时间给用户反馈

### 第二层：异步业务校验

例如：

- 学号唯一性

它必须请求服务端，因为“当前学号是否存在”是服务端数据问题，不是纯前端规则问题。

### 第三层：提交后的服务端最终校验

即使前端已经校验过，后端仍然必须再检查：

- 防止绕过前端
- 防止并发提交导致的数据竞争
- 保证数据一致性

### 为什么这是典型面试点

因为它能体现你是否理解：

- 用户体验
- 数据安全
- 前后端职责边界

## 16.7 为什么 `checkStudentNoUnique()` 当前是通过列表接口实现的

前端的 `checkStudentNoUnique()` 实际上是：

- 走 `fetchStudents()`
- 再在结果里本地查重

而不是直接请求一个独立唯一性检查接口。

### 这是一种什么思路

这是“复用已有列表接口”的轻量做法。

优点：

- 少写一个接口也能完成功能
- 项目初期开发更快

缺点：

- 不是最精准的接口语义
- 请求数据可能比实际所需更多
- 当学生数量变大时效率较低

### 但注意

后端其实已经补了 `/students/check-no` 的能力，只是前端暂时没切过去。

这正是一个很好的“演进点”：

- 当前实现能用
- 下一步可以改成专门校验接口

这类“知道当前方案不完美，并清楚下一步怎么演进”的表达，面试里很加分。

## 16.8 为什么 `course_ids` 用 JSON 数组而不是关联表

这个问题非常适合面试，因为它涉及设计权衡。

### 当前做法

在 `students` 表中直接存：

```json
[1, 3, 5]
```

### 这样做的优点

- 实现简单
- 开发快
- 很适合作业项目或小型原型
- 后端写入逻辑直观

### 缺点

- 不能高效用 SQL 做关系查询
- 不能直接做 join
- 按课程过滤时需要在内存中处理
- 数据约束弱于中间表

### 如果升级到生产项目

推荐改成：

```text
students
courses
student_courses
```

中间表的优势：

- 关系表达更标准
- 查询更灵活
- 更适合统计和扩展
- 数据一致性更好

### 面试回答模板

“当前项目规模较小，为了降低建模复杂度和提高开发效率，我先用 JSON 数组存课程关系。这个方案的缺点是查询关系不够优雅，尤其是按课程筛选时效率不高。如果进入更真实的业务场景，我会拆成中间表 `student_courses`，把关系查询和约束交给数据库层处理。”

## 16.9 为什么 Koa 的静态托管要做 SPA 回退

服务端在处理非 `/api` 的 GET 请求时：

- 先看请求的资源文件是否存在
- 如果存在，直接返回静态文件
- 如果不存在，回退到 `index.html`

### 为什么必须这样做

因为前端使用的是浏览器路由：

- `/dashboard`
- `/courses`
- `/students`

用户如果直接刷新 `/courses`：

- 浏览器会把它当成一个真实 URL 去请求服务器
- 服务器如果不回退到 `index.html`，就会返回 404

### 面试常见问法

“为什么 React Router 项目部署后刷新页面会 404？”

答案就是：

- 浏览器刷新时请求的是服务端 URL
- 而前端路由只有在 `index.html` 启动后才能接管
- 所以服务端必须把未知前端路由回退到 `index.html`

---

## 16.10 为什么要对静态资源做目录穿越防御

在 `static.js` 中，路径经过：

- `join`
- `normalize`
- 再校验是否仍然以 `DATA_ROOT` 开头

这是为了防止类似这样的访问：

```text
/api/static/../../src/index.js
```

如果不做校验，攻击者可能读到：

- 服务器源码
- 配置文件
- 数据文件

### 这类问题在面试里属于什么层次

属于“你有没有安全意识”的问题。

哪怕项目本身不是高安全系统，能想到目录穿越防御，说明你不是只会堆功能。

## 18. 性能、可维护性与扩展性分析

## 18.1 当前性能表现在哪些地方已经足够

对这个项目而言，当前性能在这些方面是足够的：

- 列表数据量小
- SQLite 单机读写开销低
- Zustand 状态量不大
- 图表数据量非常有限
- Markdown 内容较短

所以很多实现虽然不是“最强架构”，但在这个场景里完全合理。

这是面试里很重要的一种思维：

- 设计要匹配场景
- 不要过度设计

---

## 18.2 当前性能瓶颈会出现在哪里

如果数据量扩大，优先出现问题的点通常是：

### 1. `course_ids` JSON 数组查询

按课程筛选学生时需要先取出数据再内存过滤。

### 2. `updateCourseCounts()`

每次学生增删改都要遍历所有课程和所有学生，数据量大时写操作会慢。

### 3. Markdown 解析器

如果内容特别长，自写解析器会在前端做更多字符串处理。

### 4. 学号唯一性前端预检查

如果继续走列表接口查重，效率会越来越低。

---

## 18.3 如果要做性能优化，可以怎么做

### 后端

- 用中间表替代 JSON 数组关系
- 用 SQL 聚合替代内存遍历
- 对热点查询增加索引
- 使用更适合多用户场景的数据库

### 前端

- 为搜索增加防抖
- 将 query 同步到 URL，便于恢复页面状态
- 长列表使用虚拟滚动
- Markdown 解析结果可缓存

---

## 18.4 维护性分析

这个项目维护性较好的地方：

- 前后端职责比较清晰
- store 分模块拆分
- API 封装统一
- 文件结构规范
- 页面风格统一
- 现在又补上了详细注释

维护性一般的地方：

- 部分逻辑仍偏作业式直写，没有进一步抽象
- 学生和课程页面有部分重复代码
- Markdown 解析器是自写实现，未来功能扩展成本较高

### 如果继续重构

可以进一步抽象出：

- 列表页通用查询栏组件

## 19. 安全性分析

## 19.1 已有的安全措施

当前项目已经体现出一些基础安全意识：

- 密码使用 `bcrypt` 加密存储
- 登录失败不暴露用户名是否存在
- JWT 做接口鉴权
- 401 统一清理登录态
- 静态资源接口有目录穿越防御
- 静态资源接口有文件类型白名单

---

## 19.2 仍然存在的安全边界

### 1. JWT 密钥硬编码

当前 `JWT_SECRET` 写在源码里。

这在学习项目里可以接受，但真实项目中应该：

- 放到环境变量
- 区分开发与生产环境

### 2. 没有 refresh token 机制

当前 token 过期后只能重新登录。

### 3. 没有登录失败限流

如果是生产系统，应该考虑：

- 登录失败次数限制
- IP 限流
- 验证码

### 4. Markdown 渲染边界有限

当前自写渲染器功能有限，如果以后支持更复杂 HTML/Markdown，需要更加重视 XSS 风险。
