# 本轮功能改造对比学习文档

## 1. 改造总览

本轮主要围绕两个方向升级：

```txt
权限体系增强
学习总结模块重构
```

改造前，项目更像一个基础后台：

- 登录后固定展示菜单
- 权限主要靠前端简单控制
- 角色权限写死在配置中
- 学习总结是单个静态 Markdown 文件

改造后，项目更接近真实后台：

- 菜单、路由、按钮、接口都有权限控制
- admin 可以管理教师和学生账号
- teacher/student 权限可在页面中配置
- admin 固定拥有全部权限，不允许被误改
- 用户可以修改自己的密码
- 学习总结变成当前用户自己的多篇内容
- 总结支持新增、分页、查看、编辑、删除、Markdown 预览和代码高亮

## 2. 权限体系对比

### 改造前

权限比较简单：

```txt
admin / teacher / student 固定写在代码里
前端根据 permissions 控制部分菜单和按钮
角色权限不能在页面中修改
```

主要问题：

- 不够像真实后台
- 面试时只能说“做了权限控制”，不能说“做了权限配置”
- 角色权限调整需要改代码

### 改造后

新增了数据库权限模型：

```txt
roles
permissions
role_permissions
```

现在权限来自数据库：

```txt
用户 role
  ↓
roles 表
  ↓
role_permissions 表
  ↓
permissions 表
  ↓
返回 permissions 数组
```

前端根据权限控制：

- 侧边栏菜单
- 页面路由
- 页面按钮
- 表格操作列

后端根据权限控制：

- 接口是否允许访问

## 3. admin 权限保护

### 问题

一开始角色权限页允许修改 admin 权限。

这会出现严重问题：

```txt
admin 取消自己的 roles:updatePermissions
  ↓
保存成功
  ↓
admin 失去角色权限页面访问能力
  ↓
无法再把权限加回来
```

这就是“管理员把自己锁死”的问题。

### 当前方案

admin 权限固定为全部权限。

前端：

- admin 卡片只读
- 显示“固定全权限”
- Checkbox 禁用
- 不显示保存按钮

后端：

- `getPermissionsByRole('admin')` 永远返回全部权限
- 修改 admin 权限接口直接拒绝
- 修改 admin 账号角色也会被拒绝

这样即使有人绕过前端直接调接口，也不能修改 admin 的最高权限。

## 4. 账号管理对比

### 改造前

账号管理只支持：

```txt
查看账号
修改账号角色
```

并且角色选择里包含 admin，存在误操作风险。

### 改造后

账号管理支持：

```txt
查看账号
新增教师/学生账号
修改教师/学生账号角色
删除教师/学生账号
```

限制规则：

```txt
admin 账号不能删除
admin 账号角色不能修改
只能新增 teacher/student
只能删除 teacher/student
新增账号初始密码为 123456
```

前端文件：

```txt
client/src/pages/AccountsPage.tsx
```

后端接口：

```txt
GET    /api/system/users
POST   /api/system/users
PATCH  /api/system/users/:id/role
DELETE /api/system/users/:id
```

## 5. 修改密码功能

### 改造前

用户登录后不能修改自己的密码。

### 改造后

右上角用户菜单新增：

```txt
修改密码
```

用户需要输入：

```txt
原密码
新密码
确认新密码
```

后端校验：

- 原密码必须正确
- 新密码至少 6 位
- 保存时重新 bcrypt 加密

接口：

```txt
PATCH /api/auth/password
```

相关文件：

```txt
client/src/layouts/AppShell.tsx
server/src/routes/auth.js
```

## 6. 角色权限页面对比

### 改造前

角色权限页直接在页面中写了大量重复 JSX：

```txt
admin 卡片一套
teacher 卡片一套
student 卡片一套
```

问题：

- 页面代码很长
- admin 和普通角色结构重复
- 保存权限没有二次确认

### 改造后

抽出了组件：

```txt
client/src/components/RolePermissionCard.tsx
```

`RolesPage` 只负责：

- 拉取角色权限数据
- 按模块分组权限
- 保存角色权限

`RolePermissionCard` 负责：

- 展示角色信息
- 展示权限分组
- 勾选权限
- 保存按钮
- admin 只读模式
- 保存二次确认

这样页面职责更清楚，也更适合学习组件拆分。

## 7. 学习总结模块对比

### 改造前

学习总结来自一个静态文件：

```txt
server/data/summary.md
```

所有用户看到同一份内容。

前端只有展示：

```txt
SummaryPage -> fetchSummary -> MarkdownRenderer
```

问题：

- 不是用户自己的数据
- 不能新增多篇
- 不能编辑
- 不能删除
- 没有列表分页

### 改造后

新增数据库表：

```txt
learning_summaries
```

字段：

```txt
id
user_id
title
content
created_at
updated_at
```

每条学习总结都绑定当前登录用户：

```txt
WHERE user_id = ctx.state.user.id
```

所以用户只能操作自己的总结。

## 8. 学习总结接口对比

### 改造前

只有：

```txt
GET /api/summary
```

返回单篇 Markdown 内容。

### 改造后

变成完整 CRUD：

```txt
GET    /api/summary       分页获取自己的总结列表
GET    /api/summary/:id   查看自己的总结详情
POST   /api/summary       新增自己的总结
PUT    /api/summary/:id   编辑自己的总结
DELETE /api/summary/:id   删除自己的总结
```

并且都需要当前用户权限：

```txt
summary:view
summary:create
summary:update
summary:delete
```

## 10. Markdown 高亮优化

### 改造前

- 没写语言时高亮弱
- 没有行号

### 改造后

- 默认按 JavaScript 高亮

- 显示行号
- 长行自动换行
- 保留复制按钮

例如这些语言都能更好识别：

```txt
js -> javascript

sh -> bash
md -> markdown
json5 -> json
```


## 13. 后端数据隔离

学习总结最关键的一点是：

```txt
用户只能操作自己的总结
```

后端所有查询都带：

```sql
WHERE user_id = ?
```

新增时写入：

```js
ctx.state.user.id
```

查看、编辑、删除时也同时校验：

```txt
summary id
user_id
```

所以即使用户猜到别人的总结 ID，也无法访问。

## 15. 学习重点

建议重点理解这几件事：

```txt
1. 为什么 admin 权限必须固定全权限
2. 为什么前端权限控制不能替代后端鉴权
3. 为什么学习总结必须用 user_id 做数据隔离
4. 为什么列表页需要 query + draftKeyword 分离
5. 为什么新增/编辑弹窗要复用同一个 submitForm
6. 为什么角色权限保存需要二次确认
7. 为什么把 RolePermissionCard 抽出去能降低页面复杂度
```
