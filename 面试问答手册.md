# 面试问答手册

## 1. 使用说明

这份文档不是项目说明书，而是专门面向面试场景整理的问答手册。

阅读目标有三个：

- 帮你知道面试官可能问什么
- 帮你理解每个问题背后真正想考察什么
- 帮你准备一套能直接说出口的话术

建议使用方式：

1. 先通读整份文档，建立答题框架
2. 再对照 [项目技术文档.md](file:///d:/DeskTop/react-project/course/项目技术文档.md) 和源码理解细节
3. 最后挑重点问题做口头复述练习

## 2. 项目总述类问题

## 2.1 先简单介绍一下你的这个项目

### 面试官想听什么

- 你能不能用一句话说清项目定位
- 你是否能快速提炼系统主干
- 你是不是只会按页面流水账描述

### 参考回答

“这是一个前后端分离的在线学习管理后台，前端使用 React、TypeScript、Zustand 和 React Router，后端使用 Koa 和 SQLite。项目包含登录认证、课程管理、学生管理、工作台统计图表和 Markdown 学习总结展示。我在这个项目里重点做了统一请求封装、路由守卫、状态管理、后端 CRUD 与统计聚合，以及前端打包后由 Koa 统一静态托管的完整闭环。”

### 回答要点

- 先说项目类型
- 再说技术栈
- 再说功能模块
- 最后说你负责或最有价值的部分

### 可追问点

- 你觉得这个项目最大的亮点是什么
- 它和普通 CRUD 项目有什么区别

## 3. 架构设计类问题

## 3.1 这个项目的整体架构是什么样的

### 面试官想听什么

- 你是否理解前后端分层
- 你是否能描述数据流

### 参考回答

“整体上是一个典型的前后端分离架构。前端负责页面、交互、状态和表单校验，后端负责认证、数据读写、统计聚合和静态资源服务。前端页面不直接写请求细节，而是通过 `api.ts` 和 `request.ts` 两层封装访问后端；业务页面再通过 Zustand store 管理状态。后端用 Koa 路由承接请求，最终落到 SQLite。整个数据流是：用户操作触发页面事件，页面调用 store，store 调用 API，请求到后端，后端查询数据库，再把统一结构的结果返回给前端更新视图。”

### 延伸回答

如果面试官继续问，你可以补一句：

“我比较强调的是职责边界清晰，页面不直接操作 axios，store 不直接操作 DOM，后端路由不直接处理前端展示逻辑。”

---

## 3.2 为什么前端要用 Zustand，不用 Redux

### 面试官想听什么

- 你是不是会做技术选型
- 你是否知道 Zustand 和 Redux 的边界

### 参考回答

“因为这个项目虽然有多个业务模块，但整体状态复杂度还没有到 Redux 那种需要严格 action/reducer 体系的程度。Zustand 更轻量，样板代码更少，特别适合这种中小型后台项目。像登录态、列表查询状态、表单弹窗状态、加载态这些都可以很自然地放在 store 里管理。如果后续项目规模继续变大，比如权限模型更复杂、需要更强的调试能力或团队规范统一，我会考虑 Redux Toolkit。”

### 回答加分点

- 不要说 Redux 不好
- 要说“当前场景下 Zustand 更合适”

---

## 3.3 为什么请求要拆成 `request.ts` 和 `api.ts`

### 参考回答

“我是有意做了两层拆分。`request.ts` 负责底层通信能力，比如 Axios 实例、token 注入、统一错误处理和响应解包；`api.ts` 负责业务语义，比如登录、获取课程列表、创建学生这些具体接口。这样页面和 store 不关心请求细节，只关心我要调用哪个业务方法。这种拆法最大的好处是职责清晰，也便于后续统一调整鉴权、超时、错误处理等机制。”

---

## 3.4 为什么路由守卫要放在布局层而不是每个页面单独做

### 参考回答

“因为这个项目的业务页面都共享同一个后台布局，所以把鉴权和重定向放在受保护布局层是最自然的。这样所有页面都自动继承同一套登录校验逻辑，不需要每个页面重复写。如果放在页面里，每个页面都要自己判断 token 和 user，重复且容易漏。放到布局层以后，导航、错误展示和用户信息展示也能一起集中处理。”

### 可继续展开

- 登录页与业务页天然是两套入口
- 受保护布局就是业务页的统一容器

---

## 4. 鉴权与登录态问题

## 4.1 你的登录功能是怎么做的

### 参考回答

“登录功能分前后端两部分。前端登录页收集用户名和密码，提交给 `/api/auth/login`。后端收到后先查用户，再用 `bcrypt` 校验密码，成功后用 `jsonwebtoken` 签发 JWT，并返回 token 和用户信息。前端收到后会把 token 和 user 持久化到 localStorage，同时更新 auth store，之后所有请求都会由 Axios 拦截器自动带上 `Authorization: Bearer <token>` 请求头。”

---

## 4.2 为什么你要做 `/auth/me` 接口

### 参考回答

“因为本地有 token 不等于登录态一定有效。用户刷新页面时，前端只能拿到 localStorage 中的 token，但它可能已经过期或者无效。所以我在应用冷启动时会调用 `/auth/me`，由后端验证 token 是否有效，并返回当前用户信息。这样前端恢复的是‘被服务端确认过的登录态’，而不是仅仅依赖本地缓存。”

### 面试官最想听的关键词

- 冷启动恢复登录态
- token 有效性校验
- 不能只信 localStorage

---

## 4.3 为什么登录失败时用户名不存在和密码错误都返回同一句提示

### 参考回答

“这是一个安全考虑。无论用户名不存在还是密码错误，都返回‘用户名或密码错误’，避免向外暴露系统里哪些账号是存在的。否则攻击者可以通过接口不断探测用户名存在性，这属于常见的安全细节。”

---

## 4.4 如果 token 过期了，前端怎么处理

### 参考回答

“我把这件事收敛在请求层处理。Axios 响应拦截器会统一判断后端响应的 `code` 或 HTTP 状态码是不是 401，如果是，就清理本地登录态并抛出统一错误。然后路由守卫在下一次渲染时发现没有合法登录态，就会重定向到登录页。这样业务页面不需要每个接口自己去写 401 处理。”

---

## 4.5 `initializePromise` 是干什么的

### 参考回答

“它是为了解决应用冷启动阶段重复初始化的问题。比如多个组件可能在启动时都依赖登录态，如果每个地方都触发 `initializeAuth()`，就可能并发请求多次 `/auth/me`。我用一个共享的 Promise 把这次初始化过程缓存起来，后续调用直接复用同一个 Promise，等它结束后再清空。这属于一个比较轻量的并发控制方案。”

### 这是个加分点

如果你主动讲到这个细节，通常会让面试官觉得你不是只会写功能。

## 5. 状态管理与页面交互问题

## 5.1 为什么课程页和学生页都用统一 `query` 管理筛选条件

### 参考回答

“因为分页、搜索、筛选本质上都属于列表查询状态。如果把这些状态拆得到处都是，刷新列表时就很难保证口径一致。所以我把它们统一收敛到一个 `query` 对象里，页面所有查询动作都通过 `updateQuery()` 更新这个对象，然后再调用刷新。这样可读性更好，也方便以后把 query 同步到 URL。”

---

## 5.2 为什么还要单独维护 `draftKeyword`

### 参考回答

“因为输入框里的内容和真正提交给接口的查询关键词不是一回事。用户输入过程中只是草稿，不应该每敲一个字就请求后端。所以我用 `draftKeyword` 保存输入中的值，只有在用户按回车或点击搜索时，才把它同步到 `query.keyword`，再刷新列表。这是典型的输入态和提交态分离。”

---

## 5.3 为什么要用 `useShallow`

### 参考回答

“因为课程页和学生页从 Zustand store 里取出的字段很多，如果直接返回一个对象，每次 store 更新都可能因为对象引用变化导致页面重渲染。`useShallow` 会对这些字段做浅比较，只有真正变了才更新，能减少无意义渲染。虽然这个项目规模不大，但这是比较好的状态订阅习惯。”

---

## 5.4 为什么删除后要重新计算页码

### 参考回答

“这是为了处理删除最后一条数据导致的空页问题。比如当前在第 3 页，正好只剩 1 条数据，删除后第 3 页就不存在了。如果不处理，用户会看到一个空白页。所以我做了 `pageAfterDelete()`，根据总数和 pageSize 自动回退到有效页码。”

### 可补一句

“这是典型的列表页体验细节，虽然不复杂，但很体现对用户操作流程的考虑。”

---

## 5.5 为什么学生删除按钮只让当前行 loading，而不是整个页面 loading

### 参考回答

“因为删除是一个局部动作，不应该让整个页面都被阻塞。我在 store 里维护了一个 `deletingId`，只给当前删除的那一行按钮显示 loading。这样用户可以明确知道是哪条数据在处理，交互体验会更自然。”

---

## 6. 表单与校验问题

## 6.1 你为什么同时做前端校验和后端校验

### 参考回答

“前端校验是为了用户体验，比如必填、格式错误可以即时提示；后端校验是为了数据安全，因为前端永远不能被完全信任，用户可以绕过页面直接调接口。所以我把它设计成两层：前端尽早提示，后端最终兜底。”

---

## 6.2 学号唯一性校验是怎么做的

### 参考回答

“当前前端是在失焦时做一次异步预校验，避免用户到提交时才发现学号重复。后端在保存时还会再次检查学号唯一性，作为最终保证。也就是说它不是只靠前端验证，而是前后端双重保证。这个设计既兼顾体验，也兼顾数据正确性。”

### 如果被问当前实现是否最优

可以补一句：

“当前前端预校验还是走已有列表能力去做的，作为项目规模下的快速实现是够用的；如果继续优化，我会切换成后端独立的唯一性检查接口。”

---

## 6.3 为什么课程多选要用自定义校验

### 参考回答

“因为课程字段是一个数组，不是普通字符串或数字，不能只靠简单的 required。我要校验的是‘至少选择一门课程’，所以用了 `validator` 自定义规则，根据数组长度决定是否通过。这样表达业务规则更准确。”

---

## 6.4 为什么弹窗关闭后要销毁内容

### 参考回答

“我这里使用 `destroyOnHidden`，主要是为了避免表单状态残留。因为课程和学生弹窗既承担新增也承担编辑，如果不销毁，关闭再打开时可能带着旧值或者旧校验状态。销毁后重新挂载，配合默认值或详情回填，状态会更干净。”

---

## 7. 后端与数据库问题

## 7.1 为什么后端选择 Koa，而不是 Express

### 参考回答

“这个项目里我更看重的是轻量、清晰和中间件组合能力。Koa 本身很简洁，异步中间件模型也比较直观，适合这种体量不大但希望结构清楚的项目。Express 当然也能做，但在这个项目里 Koa 足够，而且写起来更干净。”

### 注意

不要说 Express 过时或者不好，这种说法容易减分。

---

## 7.2 为什么选择 SQLite

### 参考回答

“因为这个项目是一个中小型学习/演示型后台系统，SQLite 的部署成本非常低，不需要额外安装数据库服务，配合 better-sqlite3 可以非常快地把数据层打通。它特别适合作业、原型和本地演示。如果进入更真实的多用户场景，我会考虑迁移到 MySQL 或 PostgreSQL。”

---

## 7.3 为什么 `better-sqlite3` 用同步 API 也可以接受

### 参考回答

“因为这个项目访问量和数据量都比较小，而且运行在单机场景中，同步 API 的性能和可维护性是可以接受的。同步写法能让代码更简洁，减少很多回调或异步控制复杂度。当然，如果系统并发量上来，或者要部署成更真实的多用户服务，我会重新评估数据库和访问方式。”

---

## 7.4 学生和课程为什么不建中间表，而是把课程 ID 存成 JSON 数组

### 参考回答

“这是一个基于项目规模的取舍。当前项目的目标是尽快搭完整功能链路，所以我先用 `course_ids` 的 JSON 数组来表达学生和课程的关系，这样实现快、结构简单、对小数据量也够用。缺点我也是清楚的，比如按课程过滤和关系统计不够优雅。如果项目继续演进，我会拆成 `student_courses` 中间表，把关系约束和查询交给数据库层。”

### 这题的重点

你要表达的不是“当前方案完美”，而是：

- 我知道它为什么这么做
- 我知道它的问题
- 我知道下一步如何升级

---

## 7.5 `student_count` 为什么做成冗余字段

### 参考回答

“因为课程列表和工作台图表都会频繁读取选课人数，如果每次都动态聚合会增加查询复杂度。所以我把 `student_count` 作为冗余字段放在课程表里，在学生增删改时同步回写。这样读会更快，代价是写的时候要多做一步维护。这本质上是用写复杂度换读性能。”

### 可继续说

“在更复杂的系统里，也可以用事件、触发器或定时任务来维护这类冗余统计。”

---

## 7.6 `parseStudentPayload()` 为什么值得讲

### 参考回答

“因为它体现了后端参数清洗和校验的集中化。新增和编辑都走这个函数，它会统一做文本归一化、回退旧值、课程数组规范化、手机号和邮箱校验、状态合法性校验等。这样逻辑不会分散在多个接口分支里，后续维护和扩展会更清晰。”

---

## 8. 工作台与图表问题

## 8.1 工作台的数据为什么不让前端自己算

### 参考回答

“因为工作台展示的是统计结果，不是原始数据。让后端直接返回统计后的结构更合理，一方面后端更靠近数据源，另一方面前端只需要渲染即可，逻辑更轻。比如课程分类分布、学生状态分布和近 7 天学习活跃度，这些都更适合在服务端聚合。”

---

## 8.2 为什么图表要封装 `ChartContainer`

### 参考回答

“ECharts 的初始化、更新和销毁都属于通用生命周期逻辑，如果每个图表都自己写，会有大量重复代码，也容易出现 resize 和 dispose 处理不一致的问题。所以我封装了一个 `ChartContainer`，业务图表只负责生成 option，实例生命周期都由容器统一管理。”

---

## 8.3 为什么 `setOption` 使用 `notMerge: true`

### 参考回答

“因为图表在切换不同数据或配置时，如果使用默认合并行为，可能会残留旧的 series、legend 或坐标轴配置。`notMerge: true` 可以让每次更新都用最新配置完整覆盖，减少图表状态污染。这是 ECharts 项目里一个很常见的稳定性处理。”

---

## 9. Markdown 与内容渲染问题

## 9.1 为什么自己实现 Markdown 渲染器，而不是直接用第三方库

### 参考回答

“因为这个项目对 Markdown 的需求比较有限，主要就是标题、段落、列表、代码块、图片和链接。我自己实现一个轻量版，一方面可以减少依赖，另一方面也能更深入理解 Markdown 解析和代码高亮的思路。当然我也清楚它的边界，如果项目需要更复杂的 Markdown 语法或更完整的安全策略，我会切换到成熟库。”

---

## 9.2 你的 Markdown 渲染器是怎么工作的

### 参考回答

“它分成两步。第一步是按行扫描，把内容解析成 `Block` 结构，比如 heading、paragraph、list、code、image；第二步再根据不同 Block 类型映射成 React 节点。行内的加粗、代码、图片和链接则通过正则做二次解析。代码高亮不是完整编译器级实现，而是通过关键字和模式匹配做轻量着色，满足当前项目展示需求。”

---

## 9.3 这种自写 Markdown 渲染器有什么风险

### 参考回答

“最大的边界有两个。第一是语法支持有限，复杂嵌套和边缘语法可能不支持。第二是安全边界有限，如果以后支持更复杂的 HTML 混排，就要更认真考虑 XSS 问题。所以当前实现适合学习和中小场景，但如果进入更真实的内容平台，我会换成熟库并补充安全策略。”

---

## 10. 静态托管与部署问题

## 10.1 为什么刷新前端路由页面会 404，你是怎么解决的

### 参考回答

“因为前端用的是浏览器路由，像 `/courses`、`/students` 这些路径刷新时，浏览器会直接请求服务器。如果服务器不知道这是前端路由，就会返回 404。我在 Koa 里加了 SPA 回退逻辑：对于非 `/api` 的 GET 请求，先判断资源文件是否存在，不存在就统一返回 `index.html`，让前端路由接管。”

---

## 10.2 为什么开发环境要配置 Vite 代理

### 参考回答

“主要是为了两件事。第一，避免开发时前后端端口不同导致的跨域问题。第二，保持请求地址一致，无论开发还是打包部署，前端都只写 `/api/...`，不用在业务代码里写死后端地址，这样切换环境更方便。”

---

## 10.3 为什么静态资源接口要做白名单和路径校验

### 参考回答

“因为这个接口允许客户端传入路径，如果不做校验，很容易出现目录穿越风险，比如通过 `../` 访问服务器不该暴露的文件。所以我一方面校验路径最终必须落在 `DATA_ROOT` 内，另一方面只允许白名单中的图片类型通过，避免接口变成任意文件下载入口。”

---

## 11. 工程化与代码质量问题

## 11.1 这个项目做了哪些工程化处理

### 参考回答

“我做的工程化处理主要有几类。第一是请求层统一封装，解决 token 注入和错误处理。第二是状态分层和模块拆分，课程、学生、工作台、总结各有自己的 store。第三是统一响应结构，前后端接口解析非常稳定。第四是 lint 规则，至少把 TypeScript、React Hooks 和基础代码规范接入了。第五是现在我又补了详细注释和技术文档，方便项目交接和学习。”

## 12. 性能与优化问题

## 12.1 这个项目的性能瓶颈可能在哪里

### 参考回答

“如果数据规模继续扩大，优先会出问题的地方有几个。第一是学生课程关系当前存在 JSON 数组里，按课程筛选需要在内存里处理。第二是 `updateCourseCounts()` 每次学生变动都会遍历所有课程和学生，写操作成本会上升。第三是学号预校验当前复用了列表接口，不是最经济。第四是 Markdown 渲染器在长文档下会有更多字符串处理开销。”

---

## 12.2 如果让你优化，你先做什么

### 参考回答

“我会先做三步。第一，把学生和课程关系改成中间表，这是最关键的数据层优化。第二，把学号唯一性校验切到专门接口，减少无效数据查询。第三，针对统计数据和关系查询补索引，必要时再评估数据库从 SQLite 迁移到更适合多用户并发的方案。”

---

## 12.3 为什么说这个项目没有过度设计

### 参考回答

“因为我在设计时是按项目规模来取舍的。比如状态管理选 Zustand 而不是 Redux，数据库选 SQLite，关系字段先用 JSON 数组，Markdown 解析先做轻量实现。这些都不是最重的方案，但在当前项目体量下是合理且高性价比的。更重要的是我知道它们的边界，也预留了后续演进方向。”

---

## 13. 安全问题

## 13.1 这个项目有哪些安全意识体现

### 参考回答

“有几个基础安全点。第一，密码不是明文存储，而是用 bcrypt 加密。第二，登录失败统一提示，不暴露用户名是否存在。第三，JWT 鉴权失败会统一返回 401。第四，静态资源接口做了目录穿越校验和类型白名单。虽然项目不是高安全系统，但我尽量把该有的基础边界考虑进去了。”

---

## 13.2 这个项目还存在哪些安全边界

### 参考回答

“当前仍然有几个边界。比如 JWT 密钥是硬编码的，真实项目里应该用环境变量；没有 refresh token 机制，token 过期后只能重新登录；没有登录失败限流；Markdown 渲染器在更复杂场景下需要进一步考虑安全问题。所以我会把它看成一个完整但偏学习型的实现，而不是已经达到生产标准的安全体系。”

---

## 16. 最后建议

真正面试时，不要试图把每一条答案原封不动背出来。

更好的方式是：

- 先记结构
- 再记关键词
- 最后用自己的话说出来

你至少要确保自己能稳定讲清下面这些问题：

- 项目整体架构是什么
- 登录态恢复是怎么做的
- 为什么选 Zustand
- 为什么列表页要统一 query
- 为什么 `course_ids` 当前这样设计
- 工作台统计为什么放后端做
- 为什么刷新前端路由会 404，怎么解决
- 项目还能怎么优化

如果这几个问题你都能讲顺，这个项目在面试里的表现就会明显上一个台阶。
