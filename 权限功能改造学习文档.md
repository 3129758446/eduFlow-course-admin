# 权限功能改造学习文档

## 1. 本次改造做了什么

本次改造把项目从“固定角色权限”升级成了“可配置角色权限”。

新增能力：

- 登录后根据角色加载权限
- 侧边栏菜单按权限动态显示
- 页面路由按权限拦截
- 页面按钮按权限显示或隐藏
- 后端接口按权限返回 403
- admin 可以查看账号并修改账号角色
- admin 可以查看角色并勾选角色拥有的权限
- 角色权限保存后会写入数据库

现在权限不再只写死在前端或后端代码里，而是落到了数据库表：

```txt
roles
permissions
role_permissions
```

## 3. 权限整体数据流

完整流程如下：

```txt
用户登录
  ↓
POST /api/auth/login
  ↓
后端根据 user.role 查询角色权限
  ↓
返回 user.permissions
  ↓
前端 auth-store 保存 user
  ↓
菜单、路由、按钮读取 permissions
  ↓
用户操作接口
  ↓
后端 requirePermission 再次校验
```

重点是：前端控制显示体验，后端控制真正安全。

前端隐藏按钮不能代替后端鉴权，因为用户仍然可能通过浏览器控制台或 Postman 直接请求接口。

## 4. 前端权限码

文件：

```txt
client/src/permissions.ts
```

这里统一定义权限码：

```ts
export const PERMISSIONS = {
  COURSES_VIEW: "courses:view",
  COURSES_CREATE: "courses:create",
  COURSES_UPDATE: "courses:update",
  COURSES_DELETE: "courses:delete",
} as const;
```

这样做有几个好处：

- 页面里不用到处写字符串
- 避免拼错权限码
- TypeScript 可以推导权限码类型
- 后续新增权限时入口清晰

对应类型：

```ts
export type PermissionCode =
  (typeof PERMISSIONS)[keyof typeof PERMISSIONS];
```

这表示 `PermissionCode` 只能是 `PERMISSIONS` 里的值。

## 5. auth-store 做了什么

文件：

```txt
client/src/stores/auth-store.ts
```

本次给 auth-store 增加了两个方法：

```ts
hasPermission(code)
hasAnyPermission(codes)
```

作用：

- `hasPermission` 判断是否拥有某一个权限
- `hasAnyPermission` 判断是否拥有多个权限中的任意一个

比如：

```ts
state.hasPermission(PERMISSIONS.COURSES_CREATE)
```

适合控制新增按钮。

```ts
state.hasAnyPermission([
  PERMISSIONS.COURSES_UPDATE,
  PERMISSIONS.COURSES_DELETE,
])
```

适合判断是否显示整个“操作”列。

## 6. Permission 组件

文件：

```txt
client/src/components/Permission.tsx
```

这个组件用于声明式控制局部 UI。

用法：

```tsx
<Permission code={PERMISSIONS.COURSES_CREATE}>
  <Button>新增课程</Button>
</Permission>
```

如果当前用户没有 `courses:create` 权限，里面的按钮不会渲染。

它也支持多个权限：

```tsx
<Permission any={[PERMISSIONS.COURSES_UPDATE, PERMISSIONS.COURSES_DELETE]}>
  <div>操作区</div>
</Permission>
```

本项目里主要用它控制按钮。

## 7. 动态菜单

文件：

```txt
client/src/router/nav-config.tsx
```

菜单配置从 `AppShell` 里抽出来了。

每个菜单项长这样：

```ts
{
  key: "courses",
  path: "/courses",
  label: "课程管理",
  permission: PERMISSIONS.COURSES_VIEW,
}
```

`AppShell` 会根据当前用户权限过滤：

```ts
const visibleNavItems = navItems.filter((item) =>
  user.permissions.includes(item.permission),
);
```

所以没有权限的菜单不会显示。

本次新增了两个菜单：

```txt
账号管理 -> accounts:view
角色权限 -> roles:view
```

## 8. 路由权限守卫

文件：

```txt
client/src/router/RequirePermission.tsx
```

菜单隐藏只是体验，用户仍然可能手动输入 URL。

所以每个受权限控制的路由都包了一层：

```tsx
<RequirePermission code={PERMISSIONS.COURSES_VIEW}>
  <CoursesRouteElement />
</RequirePermission>
```

如果没有权限，会跳转到：

```txt
/403
```

同时会把缺失权限传过去：

```ts
state={{ requiredPermission: code }}
```

403 页面会显示：

```txt
当前角色：学生，缺少权限：roles:view
```

这样调试时比较直观。

## 10. 账号管理页

文件：

```txt
client/src/pages/AccountsPage.tsx
```

功能：

- 加载账号列表
- 加载角色列表
- 显示账号当前角色
- 修改账号角色
- 保存后调用接口更新

相关接口：

```txt
GET   /api/system/users
PATCH /api/system/users/:id/role
```

对应权限：

```txt
accounts:view
accounts:updateRole
```

页面里的核心状态：

```ts
accounts      // 账号列表
roles         // 角色列表
draftRoles    // 当前页面里选择但还没保存的角色
savingId      // 哪一行正在保存
```

`draftRoles` 的作用是让用户可以先选择角色，再点击保存，不会一选就立即提交。

## 14. 数据库初始化

文件：

```txt
server/src/database/init.js
```

本次新增表：

```sql
roles
permissions
role_permissions
app_migrations
```

`roles` 保存角色：

```txt
admin
teacher
student
```

`permissions` 保存权限字典。

`role_permissions` 保存角色拥有哪些权限。

`app_migrations` 用来记录一次性迁移是否执行过。

为什么需要 `app_migrations`？

因为项目之前已经有旧数据库，如果直接每次启动都重置权限，会覆盖你在页面里勾选保存过的结果。

所以这里只在第一次迁移时写入默认权限。

## 15. 后端系统管理接口

文件：

```txt
server/src/routes/system.js
```

新增接口：

```txt
GET   /api/system/users
PATCH /api/system/users/:id/role
GET   /api/system/roles
PUT   /api/system/roles/:code/permissions
```

这些接口都经过两层校验：

```js
authenticateToken
requirePermission(...)
```

例如修改账号角色：

```js
router.patch(
  '/users/:id/role',
  authenticateToken,
  requirePermission(PERMISSIONS.ACCOUNTS_UPDATE_ROLE),
  async (ctx) => {}
);
```

意思是：

- 先判断是否登录
- 再判断是否有修改账号角色的权限

## 16. 后端接口权限中间件

文件：

```txt
server/src/middleware/auth.js
```

新增：

```js
requirePermission(permission)
```

逻辑：

```txt
从 ctx.state.user.role 获取角色
  ↓
查询这个角色的权限
  ↓
如果没有目标权限，返回 403
  ↓
如果有权限，继续执行接口
```

这就是接口级鉴权。

## 17. 登录权限刷新

登录接口：

```txt
POST /api/auth/login
```

刷新登录态接口：

```txt
GET /api/auth/me
```

都会返回：

```ts
permissions: string[]
```

所以用户刷新页面后，权限状态不会丢失。

角色权限页还有一个细节：

如果当前用户修改的是自己的角色权限，会调用：

```ts
initializeAuth()
```

重新拉取当前用户权限，让菜单和按钮状态及时更新。

## 19. 常见问题

### 为什么前端和后端都有权限码？

前端需要权限码控制菜单、按钮、路由。

后端需要权限码控制接口安全。

小项目里前后端各维护一份可以接受。更规范的做法是抽成 shared 包，或者由后端下发权限字典。

### 为什么修改权限后要重新登录或刷新权限？

因为前端的权限保存在 `auth-store` 的 `user.permissions` 里。

如果角色权限被修改，当前用户内存里的权限需要重新拉取。

本项目在修改当前角色权限后会调用 `initializeAuth()` 刷新。

### 为什么没有做新增账号？

后端接口里已经预留了 `POST /api/system/users`，但前端第一版只做“查看账号、修改角色”。

如果后续继续扩展，可以加一个新增账号弹窗。

